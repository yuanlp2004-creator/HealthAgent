# P1 需求分析与系统设计

> 周期：2025-11-27 ~ 2025-12-20（4 周）
> 目标：把任务书的三项课题内容落成可评估的需求规格与可实施的系统设计。

## 1. 目标

- 对齐利益方（学生、指导教师、答辩评委）对功能与验收的预期
- 产出可作为后续所有阶段"单一事实源"的设计文档
- 形成数据库 ER 图、接口草案、界面原型，降低返工风险

## 2. 输入

- 任务书（[task.pdf](task.pdf)）
- 参考文献 1：百度 AI Studio 液晶屏识别
- 参考文献 2：Advanced RAG Techniques
- 指导教师的口头/书面补充意见

## 3. 任务拆解

### 3.1 需求分析（第 1 周）

- **功能性需求**
  - FR-01 用户注册/登录/个人信息管理
  - FR-02 血压计图片上传 → OCR 识别 → 字段确认 → 入库
  - FR-03 手动录入健康数据（血压、心率、体温、体重等）
  - FR-04 健康数据列表 / 图表看板 / 导出 CSV
  - FR-05 基于历史数据的趋势预测与异常提醒
  - FR-06 与健康智能体对话（RAG + LLM，附引用）
  - FR-07 医学知识库管理（后台上传/分片/向量化）
- **非功能性需求**
  - NFR-01 单次 OCR 响应 ≤ 3s（在 CPU 环境下）
  - NFR-02 单次问诊首 token 延迟 ≤ 3s
  - NFR-03 用户数据与其他用户严格隔离
  - NFR-04 所有 AI 输出带"仅供参考，非医疗诊断"声明

### 3.2 用户画像与用例（第 1 周）

- 主用户：关注自身血压的中老年人/慢病患者及其子女
- 核心用例图（需在文档中补充绘制）：拍照记录 / 查看趋势 / 咨询建议

### 3.3 系统架构设计（第 2 周）

- 分层：表示层（React）/ 接口层（FastAPI）/ 服务层（OCR/Health/RAG）/ 数据层（PG+Chroma+对象存储）
- 关键决策：
  - OCR 采用同步接口（简化）+ 图片存本地/对象存储
  - RAG 采用朴素 RAG 起步，若时间允许引入 Hybrid Search & Rerank（参考文献 2）
  - 预测采用"规则 + Prophet"双通道

### 3.4 数据库设计（第 2 周）

核心表：

| 表 | 关键字段 | 说明 |
| --- | --- | --- |
| `users` | id, username, email, password_hash, gender, birth_date, created_at | 用户 |
| `health_records` | id, user_id, record_time, systolic, diastolic, heart_rate, source('ocr'\|'manual'), image_id, note | 健康记录 |
| `images` | id, user_id, path, ocr_raw_json, created_at | 上传的血压计图片 |
| `conversations` | id, user_id, title, created_at | 问诊会话 |
| `messages` | id, conversation_id, role, content, citations_json, created_at | 会话消息 |
| `kb_documents` | id, title, source, tags, created_at | 知识库文档 |
| `kb_chunks` | id, doc_id, chunk_idx, text, vector_id | 分片（向量存在 Chroma） |

### 3.5 接口草案（第 3 周）

REST 规范，`/api/v1/...`。示例：

```
POST /auth/register             注册
POST /auth/login                登录 -> JWT
GET  /users/me                  当前用户信息
POST /ocr/bp                    上传血压计图片，返回识别结果
POST /health/records            新建记录
GET  /health/records?from&to    查询记录
GET  /health/stats              聚合指标
GET  /health/forecast           趋势预测
POST /chat/ask                  问诊（SSE 流式）
POST /kb/docs                   管理员上传知识库文档
```
