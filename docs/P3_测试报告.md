## 1. 测试范围

本阶段交付「血压计图像识别 → 字段抽取 → 后端接口」链路并完成真实样本端到端评测。测试覆盖：

| 模块 | 覆盖点 |
| --- | --- |
| `app/services/ocr/preprocess.py` | 图像解码、缩放到最长边 1920、CLAHE 对比度增强、三变体输出 |
| `app/services/ocr/extractor.py` | `parse_tokens` 正则抽数、数值范围过滤；`extract_fields` Y 排序 + bbox 高度过滤 + `sys > dia` 校验 |
| `app/services/ocr/baidu_client.py` | 百度云 access_token 缓存、`accurate` + `numbers` 接口 |
| `app/services/ocr/qwen_vl_client.py` | 通义千问 `qwen-vl-plus` 调用、JSON 解析与异常兜住 |
| `app/services/ocr_service.py` | 引擎分发（baidu / paddle），VLM 兜底：字段缺失时自动调 qwen-vl-plus 覆盖 |
| `app/api/v1/ocr.py` | `POST /api/v1/ocr/bp`：鉴权、媒体类型白名单、空/超限校验、落盘 |
| `tests/ocr_bench.py` | 4 张真实家用血压计照片端到端准确率与耗时 |

## 2. 测试方法

### 2.1 单元测试（pytest）

- **工具**：`pytest 8.3.3` + `httpx 0.27.2`（FastAPI TestClient）
- **运行命令**：

  ```bash
  cd backend
  python -m pytest
  ```

- **测试结构**：`backend/tests/test_ocr_*.py`，共 4 个文件、**20** 个 OCR 相关用例（叠加 P1 的 27 个用例 → **全量 47**）。
- **网络隔离**：`test_ocr_vlm_fallback.py` 通过 `monkeypatch.setattr` 替换 `qwen_vl_client.recognize_bp` 与内部百度/paddle 调用，**不发出任何真实请求**。
- **鉴权构造**：`test_ocr_api.py` 通过已有 `auth/register` 接口生成 access token，再用 `Authorization: Bearer` 头访问 `/api/v1/ocr/bp`。

### 2.2 真实样本基准（bench）

- **运行命令**：

  ```bash
  cd backend
  python tests/ocr_bench.py
  ```

- **脚本**：加载 `datasets/bp_images/labels.json`（4 张真实家用电子血压计照片），对每张调用 `ocr_service.recognize()` 完整链路（含百度云识别 → 字段抽取 → VLM 兜底），与人工标注 ground truth 比对。
- **指标**：字段级命中率（systolic/diastolic/heart_rate 各自 ok/n）+ 整体准确率 + 平均单张耗时。

## 4. 方案演进与对比实验

本阶段在工程实现过程中做了四次方案迭代，全部在 4 张真实样本上做过端到端实测。关键数据留档备论文对比章节使用：

| 迭代 | 方案 | 字段准确率 | 平均耗时 | 备注 |
| --- | --- | --- | --- | --- |
| 1 | PaddleOCR PP-OCRv4（CPU，通用中英） | 0/12 (0%) | 2.42 s | 对七段数码管识别能力有限 |
| 2 | PaddleOCR + 3 变体预处理（原图/CLAHE/反色） | 1/12 (8.3%) | 4.32 s | 召回略升，误识仍多 |
| 3 | 百度云 `accurate` 通用 OCR | 1/12 (8.3%) | 0.86 s | 主数字常漏识，被小字干扰 |
| 4 | 百度云 `numbers` 数字 OCR（主）+ bbox 高度过滤 | 0/12 (0%) | 0.90 s | 主数字召回偏低 |
| 5 | 百度云 `numbers` + `accurate` 合并 + 高度过滤 | 0/12 (0%) | 1.50 s | 干扰增加 |
| 6 | **百度云（主）+ 通义千问 qwen-vl-plus（兜底）** | **9/12 (75.0%)** | 2.71 s | 最终采用方案 |

**结论**：通用 OCR 对家用血压计 LCD/七段数码管识别力天然有限；引入多模态视觉大模型作为兜底，既保留了便宜、快速的百度云 API 作为第一道，又在百度云缺字段时用 VLM 的语义推理能力覆盖剩余 bad case，是工程与成本兼顾的最优解。

## 5. 缺陷与观察

| ID | 级别 | 现象 | 根因 / 处理 |
| --- | --- | --- | --- |
| D-05 | High | 001.png 识别结果 120/80/65，与真实值 138/91/102 全错 | 图片本身屏幕反光 + 部分模糊，VLM 倾向给出"合理估计"而非拒答。**缓解**：P4 前端闭环强制用户校正低置信度字段；后续扩充数据集后考虑提示工程（让 VLM 在不清晰时返回 null） |
| D-06 | Medium | `pydantic_settings` 将 `.env` 中 `CORS_ORIGINS=a,b` 当作 JSON 解析失败 | 把 `.env` 里值改为 `["a","b"]` JSON 数组形式；实际逻辑由 `field_validator` 处理，功能不受影响 |
| D-07 | Low | PaddleOCR 首次调用自动下载 ~20MB 模型，CI 里可能较慢 | 已通过 `OCR_ENGINE=baidu`（默认）绕过；若环境变量切到 `paddle`，首次调用有约 10s 冷启动 |

## 6. 风险与下一步

| 风险 | 状态 | 缓解 |
| --- | --- | --- |
| 样本量仅 4 张，75% 的准确率置信区间宽 | 已记录 | P4/P7 补齐 ≥ 50 张覆盖不同机型、光照、角度，重跑 bench 后更新报告 |
| VLM 单次调用 2-3 s，若并发高延迟叠加 | 已记录 | 仅在主通路缺字段时触发；可加 Redis 缓存 `sha256(image_bytes) -> 识别结果` |
| DashScope / 百度云断网 | 已记录 | 环境变量 `OCR_ENGINE=paddle` 切本地推理降级 |
| VLM 对极不清晰图像可能"幻觉"合理值 | 已记录 (D-05) | 前端 UI 必须让用户确认后才落库；考虑在 prompt 中强调"不确定请返回 null" |
| 图片存储当前是本地 `storage/images/`，多实例部署有数据孤岛 | 已记录 | P7 部署阶段替换为 MinIO / 对象存储 |

## 7. 结论

- P3 功能交付：图像上传 → 预处理 → OCR 双通路识别 → 字段抽取 → 接口返回，**全链路贯通**
- 单元测试：**47/47 通过**（P1 27 + P3 20）
- 真实样本准确率：**75.0%**（字段级 9/12）。相比纯 PaddleOCR（0%）与纯百度云（8%）有显著提升
- 与 `docs/03_OCR模块.md` 的 ≥90% 指标仍有差距，但样本量（n=4）偏小；**本阶段交付条件为"方案骨架 + 真实评测 + 方案选型对比"，已满足**
- 进入下一阶段（P4 健康数据管理与预测）条件：**满足**。遗留改进项（样本扩充、前端校正闭环、缓存）纳入 P4/P6 范围
