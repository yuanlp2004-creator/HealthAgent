## 4. 代码走查关键点

### 4.1 后端

- **用户隔离**：`bp_record_service.get` 同时校验 `rec.user_id == user_id`，否则统一抛 `BpRecordNotFoundError` 映射为 404；避免通过 id 探测他人数据。
- **趋势算法**：`forecast()` 先按日聚合（多次测量取均值），再对 `daily_sys` 做 window=min(7, n) 的滑动均值；趋势判据为 `(recent_avg - prev_avg) / prev_avg`，阈值 ±3%，避免短期噪声误报。
- **建表时机**：[backend/app/main.py:11](../backend/app/main.py#L11) 通过 `from app import models` 触发模型注册；`BpRecord` 经由 `router → bp_records → bp_record_service` 的导入链在应用启动期被 SQLAlchemy 登记，`Base.metadata.create_all` 正确建表。`models/__init__.py` 目前仅显式导出 `User`，未导出 `BpRecord`，运行期无影响，但建议补上以提升可读性（见第 6 节）。
- **schema 严格性**：`BpRecordCreate.source` 使用 `Literal["manual","ocr"]`，杜绝前端写入任意字符串绕过来源校验。

### 4.2 前端

- **OCR 回填 UX**：[BpRecordForm.tsx:45-67](../frontend/src/pages/BpRecordForm.tsx#L45-L67) 上传后调 `/ocr/bp`，按字段选择性填充（`!= null`），并分级提示 3/3、部分、零识别；用户仍可改表单后再提交，`source` 自动切为 `ocr` 并保留 `image_id`。
- **看板图表**：[BpDashboard.tsx:51-86](../frontend/src/pages/BpDashboard.tsx#L51-L86) 双 Y 轴区分 mmHg 与 bpm；legend 置顶（`top: 0`）+ `grid.containLabel: true`，解决此前 x 轴时间串与图例重叠的问题。
- **导航闭环**：[App.tsx:24](../frontend/src/App.tsx#L24) 根路径与 404 均重定向到 `/bp/dashboard`；[LoginPage.tsx:22](../frontend/src/pages/LoginPage.tsx#L22) 登录成功跳 `/bp/dashboard`；[AppLayout.tsx:6-11](../frontend/src/components/AppLayout.tsx#L6-L11) 顶栏菜单覆盖全部业务页面。

## 6. 已知问题 / 后续建议

1. `backend/app/models/__init__.py` 仅显式导出 `User`，未列出 `BpRecord`。运行期通过服务模块的副作用导入仍正常，但建议补齐以免后续新增模型时误判注册时序。
2. 前端缺少 `BpRecordForm / BpRecordList / BpDashboard` 的 vitest 用例，目前仅靠 TS 类型与 E2E 手测兜底。P5 建议补充 React Testing Library 级别的页面级测试。
3. `forecast` 目前采用日聚合 + 7 日滑动均值，属于经验启发式；P5 若引入 Prophet / LSTM 需重新设计接口语义（返回置信区间等）。
4. `source` 字段当前仅区分 `manual / ocr`，未覆盖「手动编辑过 OCR 结果」这一中间态；如需后续做准确率统计，可扩展为 `ocr_edited`。
